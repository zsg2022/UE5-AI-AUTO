# UE5 开发 Skill 方案

> 状态：方案讨论稿 | 2026-05-15

---

## 一、背景

当前 72 个 MCP 工具给了 AI "操作能力"（能创建 Actor、连线蓝图、编辑数据表），但缺"专业认知"。AI 不知道 GameMode 和 PlayerController 的职责分工、不知道 BT+BB+EQS 的 AI 标准管线、不知道 EnhancedInput 的正确配置方式。

**结果**：AI 能执行单步操作，但不会设计完整功能。

## 二、核心设计决策

### 🔴 不在 Skill 里写死知识

| 原因 | 依据 |
|------|------|
| UE 版本迭代快 | 5.7 (2025.11) → 5.8 Preview (2026.5.13) → 5.8 正式 (2026.6)，半年一个大版本 |
| API 频繁破坏性变更 | 5.7 就改了：TSoftObjectPtr 类型安全、GetAssetRegistryTags 签名、FindOrAddRow 移除、PCGDistanceToDensity 砍掉。写死的 API 签名 3 个月就过期 |
| 知识面不可穷尽 | Niagara 100+ 模块参数，GAS 数十个类，全写 Skill 文件会爆炸 |
| 违反项目规范 | CLAUDE.md 规定"第三方 API 须查官方文档确认签名和版本，不凭记忆" |

### 🟢 Skill 存三层

```
层级        内容                                  存放
──────────────────────────────────────────────────────
方法层      遇到X问题→查Y文档→用Z工具执行          SKILL.md
项目层      BP_/MI_/SM_ 命名前缀、目录结构           SKILL.md（项目特化）
知识层      API 签名、类继承链、参数范围              官方文档（运行时 WebFetch）
```

### 3.2 方法层：查证策略

每次 UE5 开发任务，AI 必须执行：

1. **先查后写**：调用 `WebFetch` 获取相关文档页，确认 API 签名和参数后，再调用 MCP 工具
2. **版本锁定**：查文档时指定 `?application_version=5.7`（当前项目版本）
3. **验证优先**：涉及不熟悉的 API 时，先调 `scan_reflection` 确认该类/函数在项目中存在
4. **Context7 备选**：当官方文档页面加载失败时，使用 `context7` MCP 工具查询备用文档源

### 3.3 项目层：项目约定

从现有项目代码和 CLAUDE.md 中提取：

- **资产命名前缀**：`BP_`（蓝图）、`MI_`（材质实例）、`SM_`（静态网格）、`ABP_`（动画蓝图）、`BT_`（行为树）、`BB_`（黑板）、`IA_`（输入 Action）、`IMC_`（输入映射上下文）
- **目录结构**：`/Game/Blueprints/`、`/Game/Materials/`、`/Game/Meshes/`、`/Game/UI/`、`/Game/AI/` 等
- **代码风格**：遵循 Epic C++ Coding Standard（`U`/`A`/`F`/`I`/`E`/`T` 前缀、PascalCase 类型、camelCase 变量）
- **模块依赖**：Public 放公开头文件，Private 放实现，Build.cs 明确 Public/Private 依赖

### 3.4 工作流层：常用任务模板

以下为 AI 可编排的常见任务序列（使用现有 72 个 MCP 工具）：

| 任务 | 工具序列 |
|------|---------|
| **创建第三人称角色** | bp_create → bp_add_component(SkeletalMesh) → bp_create_function → bp_create_node(EnhancedInput) → bp_compile |
| **搭建基础 AI 敌人** | bt_create → bt_add_composite(Selector) → bt_add_task → add_bb_key → create_blackboard → create_actor(AIController) |
| **创建可拾取物品** | bp_create(Actor) → bp_add_component(StaticMesh+SphereCollision) → bp_create_node(OnBeginOverlap) → bp_compile |
| **搭建 UI 主菜单** | create_widget → add_widget_to_canvas(Button+TextBlock) → bp_create_node(OnClicked) |
| **创建过场动画** | create_sequence → add_actor_to_sequencer → add_transform_track → set_keyframe |

> 模板不是写死的脚本，而是"参考序列"——AI 在执行前先查文档确认每一步的参数，再调用工具。

| 功能 | Given-When-Then |
|------|----------------|
| 查文档路线图 | Given 用户说"帮我创建一个第三人称角色" / When AI 执行 / Then AI 先 WebFetch 查阅 Gameplay Framework + EnhancedInput 文档，再调用 bp_create 等 MCP 工具 |
| 工作流模板 | Given 用户说"搭建基础 AI 敌人" / When AI 执行 / Then AI 先查 BehaviorTree 文档确认 API，再依次调用 bt_create→bt_add_composite→bt_add_task→create_blackboard→add_bb_key，最终产出可用的行为树资产 |
| 查证策略 | Given AI 需要调用一个不熟悉的 UE5 C++ API / When AI 执行 / Then AI 先 WebSearch/WebFetch 官方文档确认签名和参数，再 context7 备选，绝不凭记忆写调用 |

## 五、Skill 使用场景

| 场景 | 触发条件 |
|------|---------|
| 用户说"创建一个 UE5 的 XXX" | AI 先查 XXX 相关文档，再用 MCP 工具执行 |
| 用户说"帮我搭一个标准的 XXX 系统" | AI 查阅 Gameplay Framework 文档确认架构，再编排工具序列 |
| 用户说"这个功能怎么做" | AI 查阅对应子系统文档，回答架构方案并用 MCP 工具演示 |
| 用户说"升级到 5.8" | AI 查阅 Release Notes 和 Upgrading Guides，列出影响项 |
| 用户说"写 UE5 C++ 代码" | AI 查阅 API Reference 确认签名，查阅 Coding Standard 对齐风格 |

## 五、禁止事项 + Non-Goals

### Non-Goals（本 Skill 故意不做的事）

| 不做 | 理由 |
|------|------|
| 写死 UE5 API 签名/类继承链 | 版本迭代太快，半年过期 |
| 离线模式（无 WebFetch） | 查证策略依赖联网，离线时技能降级但不崩溃 |
| 项目模板生成 | 那是 implementation-plan-writer 的职责 |
| 代替 code-review 做 UE5 专项检查 | 那是 code-review 的职责，本 Skill 只提供 UE5 专业知识输入 |
| 生成 C++ 代码 | 那是 code-writer 的职责，本 Skill 只告诉 AI "用什么 API、去哪里查" |

### 风险与缓解

| 风险 | 概率 | 影响 | 缓解 |
|------|:---:|:---:|------|
| Epic 重构文档 URL 路径 | 中 | 高 | Skill 中放相对路径模式（`/documentation/en-us/unreal-engine/<topic>`），不硬编码完整 URL；GPT/Claude 可通过搜索找到新位置 |
| UE 版本升级后查证策略 version 参数失效 | 高 | 低 | version 参数按项目配置动态填入，Skill 中存模板 `?application_version=<当前版本>` |
| context7 服务不可用 | 低 | 中 | 降级到 WebSearch + WebFetch，不阻断流程 |

### 禁止事项

- ❌ 不凭记忆写 UE5 API 调用（必须查文档确认签名）
- ❌ 不凭记忆猜测模块依赖关系（必须查 Build.cs 或 API Reference 确认模块归属）
- ❌ 不假设 UE 版本间的 API 兼容性（版本迁移必须查 Release Notes）
- ❌ 不用 `std::vector`/`std::map` 替代 `TArray`/`TMap`（除非显式互通层代码）

## 七、与现有 Skills 的关系

| 现有 Skill | 关系 |
|-----------|------|
| `code-writer` | UE5 Skill 提供 UE5 专业知识，code-writer 负责编码执行 |
